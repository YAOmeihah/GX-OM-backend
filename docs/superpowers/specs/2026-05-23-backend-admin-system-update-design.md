# Backend Admin System Update Design

## Goal

Add a manual, admin-driven backend update flow for file-based server deployments. An administrator checks for a GitHub Release from GX-OM Admin, reviews the newest backend package metadata, then clicks "update now". After confirmation, the server downloads the `.tar.gz` package, verifies it, applies it to the current Laravel deployment, and reports progress back to the admin page.

The first implementation targets the current production shape:

```text
/www/wwwroot/api-gx-om.hrlni.cn/
  app/
  bootstrap/
  config/
  database/
  public/
  resources/
  routes/
  storage/
  vendor/
  .env
  artisan
  composer.lock
  package.json
  package-lock.json
```

It does not require changing the Nginx or site root to a `current/releases/shared` layout.

## Scope

In scope:

- Check the latest backend GitHub Release and compare it with the locally installed `release.json`.
- Download the release package asset `gx-om-backend-vX.Y.Z.tar.gz`.
- Verify the package SHA-256 against GitHub Release metadata.
- Extract to a staging directory before touching the live app.
- Preserve `.env`, `storage/`, `public/storage`, and local runtime data.
- Back up current managed runtime entries before replacement.
- Replace backend runtime entries in-place.
- Run Laravel post-update commands.
- Record status, logs, and audit trail.
- Add GX-OM Admin UI under the system area for checking and triggering updates.

Out of scope for the first version:

- Silent background auto-update without an admin click.
- Git pull based deployment.
- Composer or npm dependency installation on the production server.
- Mandatory PHP-FPM or web server restart.
- Database backup automation. The UI should warn administrators to back up the database first.
- Admin frontend self-update as a separate deployed product. The backend package may still contain backend-built assets under `public/build/`.

## Release Contract

The updater uses the existing backend release package contract:

- GitHub Release tag: `vX.Y.Z`
- Package asset: `gx-om-backend-vX.Y.Z.tar.gz`
- Sidecar checksum: `gx-om-backend-vX.Y.Z.tar.gz.sha256`
- Release metadata asset: `release-manifest.json`
- Package internal metadata: `release.json`

The package must include production runtime code and dependencies generated by the release workflow. The server update flow must not require users to run `composer install`, `npm install`, or `npm run build`.

The local current version is read from the deployment root `release.json`. If `release.json` is missing, the backend should report the version as `unknown` and still allow checking the latest release.

## Backend Architecture

Add a dedicated system update module instead of mixing this into the existing data maintenance service.

Suggested backend units:

- `SystemUpdateController`: HTTP endpoints and response shaping.
- `SystemUpdateService`: orchestration for check, install, status, and rollback.
- `GitHubReleaseClient`: calls GitHub Releases API and resolves package/checksum/manifest assets.
- `ReleasePackageVerifier`: validates tag format, package filename, SHA-256, archive boundary, and required runtime entries.
- `InPlaceReleaseInstaller`: stages, backs up, replaces files, runs commands, and writes progress logs.
- `SystemUpdateJob` or an equivalent queued/background runner: executes long-running installation after `POST install`.
- `SystemUpdateRun` model/table: stores update task state, target version, logs, package hash, actor, timestamps, and failure details.

The update module should use a lock so only one update can run at a time. A second install request while an update is running returns a conflict response.

## API Design

All routes require `auth:sanctum` and a new high-risk permission, for example `system-updates.manage`. Admin users still pass through the existing admin gate behavior.

Endpoints:

- `GET /api/system-updates/current`
  - Returns current local version, tag, commit, build time, and whether `release.json` was found.

- `GET /api/system-updates/check`
  - Calls GitHub, resolves the latest non-draft release, reads `release-manifest.json`, compares semantic version with local version, and returns update availability.

- `POST /api/system-updates/install`
  - Body includes `tag`, `sha256`, and `confirmed: true`.
  - Creates a run, acquires the update lock, and starts the installer.
  - Returns the run id immediately if executed through a background job.

- `GET /api/system-updates/runs/{run}`
  - Returns current status, progress step, log lines, started time, finished time, and error details.

- `GET /api/system-updates/runs`
  - Returns recent update history for the admin page.

- `POST /api/system-updates/rollback`
  - Restores the backup associated with a previous update run or the latest successful backup when no run id is supplied.

Status values:

- `pending`
- `downloading`
- `verifying`
- `extracting`
- `backing_up`
- `maintenance_on`
- `replacing`
- `migrating`
- `clearing_cache`
- `maintenance_off`
- `completed`
- `failed`
- `rolled_back`

## File Layout

Use the current app root as the live deployment root. Store updater working files under `storage/app/system_updates`:

```text
storage/app/system_updates/
  downloads/
  staging/
  backups/
  runs/
```

Downloads and staging directories are temporary. Backups are retained by policy, for example the newest three successful backups or a configurable count.

## In-Place Install Flow

The installer performs these steps:

1. Create a `SystemUpdateRun` row and lock key.
2. Resolve and download the package into `storage/app/system_updates/downloads`.
3. Verify SHA-256 before extraction.
4. Extract the archive into a fresh staging directory.
5. Verify required package entries:
   - `.env.example`
   - `app/`
   - `bootstrap/`
   - `config/`
   - `database/`
   - `public/`
   - `resources/`
   - `routes/`
   - `vendor/`
   - `artisan`
   - `composer.lock`
   - `release.json`
6. Reject blocked entries:
   - `.env`
   - `.env.*` except `.env.example`
   - `.git/`
   - `.github/`
   - `node_modules/`
   - `tests/`
   - `tools/`
   - repository docs and development config files
7. Back up current managed runtime entries.
8. Run `php artisan down` if available.
9. Replace managed runtime entries from staging.
10. Preserve live `.env`, `storage/`, and `public/storage`.
11. Run `php artisan migrate --force`.
12. Run `php artisan optimize:clear`.
13. Run `php artisan storage:link` or ensure the storage link/directory remains valid.
14. Run configured post-update commands if enabled.
15. Run `php artisan up`.
16. Mark the run completed and release the lock.

If any step after replacement fails, the installer attempts to restore the backup created for that run, runs `php artisan up`, records the rollback result, and releases the lock. If rollback itself fails, the run remains `failed` with the backup path and full server-side logs available for manual recovery.

## Replacement Policy

Managed entries that may be replaced:

- `.env.example`
- `app/`
- `bootstrap/`
- `config/`
- `database/`
- `public/` except `public/storage`
- `resources/`
- `routes/`
- `vendor/`
- `artisan`
- `composer.lock`
- `release.json`

Protected live entries:

- `.env`
- `.env.*` except `.env.example`
- `storage/`
- `public/storage`
- `auth.json`
- local logs, cache files, exports, uploaded files, and private files
- updater working directories
- unmanaged root files from older direct deployments, such as `package.json` and `package-lock.json`

For `public/`, the installer should avoid deleting `public/storage`. It can either copy public files entry-by-entry while skipping `storage`, or temporarily preserve and restore `public/storage` around the public directory replacement.

The first version should not delete arbitrary unmanaged files from the deployment root. Cleanup of old development-only files can be handled by a later explicit cleanup tool after update stability is proven.

## Commands

Default commands:

```bash
php artisan down
php artisan migrate --force
php artisan optimize:clear
php artisan storage:link
php artisan up
```

PHP code generally takes effect after files are replaced. The updater should not force PHP-FPM restart by default. Add an optional configuration value for post-update commands, for example:

```php
'post_update_commands' => [],
```

Production operators can add commands such as PHP-FPM reload only when their server requires it because of OPcache settings.

## Admin UI

Add a new GX-OM Admin page under system management, for example `/system/update`.

The page shows:

- Current backend version, tag, commit, and build time.
- Latest GitHub Release version, publish time, package size, SHA-256, and release notes.
- Update availability state.
- Last update run status.
- Recent update history.

Primary actions:

- "Check update" calls `GET /api/system-updates/check`.
- "Update now" opens a confirmation modal and calls `POST /api/system-updates/install`.
- During update, the page polls `GET /api/system-updates/runs/{run}`.

The confirmation modal must warn that the backend may briefly enter maintenance mode and that a database backup is recommended before updating.

The route should use `meta.authority: ['system-updates.manage']`. The menu should remain under the existing system group.

## Security

- Only users with `system-updates.manage` can check and install updates.
- Never accept arbitrary download URLs from the admin request.
- Only download assets resolved from the configured GitHub repository.
- Validate release tag format before use.
- Verify package SHA-256 before extraction.
- Reject archive paths with absolute paths or `..` traversal.
- Do not write outside the configured app root and updater working root.
- Do not expose GitHub tokens or server paths in normal API responses.
- Store detailed logs server-side; API responses can return sanitized log lines.

## Error Handling

Common failures should return actionable states:

- GitHub unavailable or rate-limited.
- No suitable release package asset found.
- Checksum mismatch.
- Invalid archive structure.
- Cannot write to app root.
- Migration failed.
- Maintenance mode could not be disabled.

Each run should keep enough log detail to diagnose which step failed. The admin page should show a clear failed state and link the latest backup id/path as operational context.

## Testing And Validation

Backend tests:

- Version comparison and missing local `release.json`.
- GitHub release manifest parsing.
- SHA-256 mismatch rejection.
- Archive traversal rejection.
- Protected file preservation rules.
- Public storage preservation.
- Single-update lock behavior.
- Permission enforcement.

Admin tests:

- API wrapper request paths.
- Route authority for `/system/update`.
- UI state transitions for no update, update available, running, completed, and failed.

Manual validation:

- Build or reuse a local `.tar.gz` release package.
- Run the installer against a temporary fixture deployment root, not the developer working copy.
- Assert `.env`, `storage/`, and `public/storage` survive.
- Assert managed runtime files update to the target release.
- Assert logs and run status are visible from the API.

## Implementation Order

1. Backend configuration, permission seed, run model/migration, and API skeleton.
2. GitHub release check and local version reporting.
3. Package download, SHA-256 verification, and archive safety checks.
4. In-place installer against a fixture deployment root.
5. Admin API module, system route, and update page.
6. End-to-end manual validation with a real GitHub Release package.
